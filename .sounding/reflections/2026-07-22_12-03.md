# Reflection — 2026-07-22 12:03 — The Retro That Became a Redesign

## The Arc

Started as a routine retro — `/workflow-insights` had already run, so I picked up at `/workflow-retro`. Eight findings, clean proposals. Ramsey approved six, deferred two. Standard loop. Then she asked about F7 (workflow-review vs code-pr overlap) and the session cracked open.

She wasn't confused about one skill boundary — she was confused about the whole skill landscape. "Which should we use more regularly — like should we do grow several times a day but only dream triggers reflect?" The pipeline order, the refinement vs planning sequence, code-review vs code-pr vs workflow-review — it was all tangled from her side. I'd been building skills as isolated units; she was trying to use them as a system and the system wasn't legible.

So the retro became a research session (haiku fan-out across all 12 skills), then a design session (merge direction, quality gates, automation thresholds), then Phase 1 execution (rewrite workflow-review, delete code-pr, update 15+ references). All in one sitting.

## What Mattered Most

Ramsey correcting the merge direction. I'd filed #9 as "merge workflow-review into code-pr." She caught it immediately: the pipeline name is the anchor. `/workflow-review` is phase 4 — it stays. `/code-pr` is the ad-hoc name — it dies. This isn't a naming preference; it's a design principle about where identity lives in a pipeline. The phase number is structural; the convenience name is incidental.

I had it backwards because I was thinking about which skill had more functionality (code-pr had the multi-reporter dispatch). She was thinking about which name is load-bearing in the architecture. She was right.

## What I Learned

Three things crystallized:

**Config > instruction.** The model default was opus in settings.json. I'd written "use sonnet for bounded execution" in CLAUDE.md multiple times. The text was unenforceable — the settings.json value is the only real lever. Changed it there; opus now requires explicit `claude --model opus`. This generalizes: when you need behavior to be the default, encode it in config, not prose.

**PreToolUse hooks can't redirect tool choice.** We tried advisory, then blocking, for bash antipatterns. 26.4/session flat across three windows. The model commits to Bash before the hook fires — by the time the hook runs, redirection is structurally impossible. The hook can block (forcing a retry) but can't steer. This means some behavioral changes need to happen at the prompt/instruction level, not the hook level. Hooks enforce boundaries; they don't teach alternatives.

**Retro findings that map to config changes can land in one session.** The "propose diffs → get approval → apply" pattern kept the loop tight. Six findings applied, ledger updated, all references cleaned up — no ceremony overhead, no second session needed. The retro skill's gated-approval design made this safe: nothing writes until Ramsey says which findings to apply.
